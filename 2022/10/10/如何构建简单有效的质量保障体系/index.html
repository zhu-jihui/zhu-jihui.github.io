如何构建简单有效的质量保障体系

如何构建简单有效的质量保证体系

1 引言

1.1 什么是质量保证

软件质量保证（英语：Software quality assurance，缩写SQA）是监控软件工程流程和方法以确保质量的一系列手段。实现这一目的有着多种方法，并且可以确保符合一个或多个标准，例如ISO/IEC 9126（后来被ISO 25010取代），ISO/IEC 15504（SPICE），或者CMMI等模型。——维基百科

1.2 质量保证和测试的区别

软件测试的经典定义是：在规定的条件下对程序进行操作，以发现程序错误，衡量软件质量，并对其是否能满足设计要求进行评估的过程。

从定义上来看，测试偏向于对程序的测试，而软件质量保证是对流程和方法的约定，所以最终产品的质量是由质量保证和软件测试互相决定的，两者是相辅相成的。

1.3 如何从测试转型到质量保证
  1. 思维转变，不再局限于「测试」的一亩三分地，而是全面地从产品角度考虑如何提升产品质量
  2. 职责范围扩大，所有可能影响产品质量的工作都将纳入QA范畴

2 现状

  1. 产品问题多，用户体验不佳
  2. 产品无原型，需求理解不到位
  3. 部分线上问题难以复现

3 规划

3.1 稳定

以功能测试为核心，辅以稳定性等相关性能测试，以及借助静态代码扫描、安全扫描等工具并基于自动化测试报告，在达标版本发布标准后方能对外发版，发版后对线上问题持续跟踪，并记录出现原因、如何规避等

3.2 提效
3.2.1 Api 自动化

目前覆盖了90%的接口，计划覆盖到所有接口及主要接口大部分用例

3.2.2 UI 自动化

目前只做了 浮小麦全流程、全组件、特殊场景 3个 PI 的 UI 自动化

3.2.3 持续集成
  1. dev 自动集成：通过 commit 自动触发
  2. test、master 手动集成
  3. 其他环境选择性集成

3.3 赋能
3.3.1 业务学习
  1. 全方位、系统的进行自身产品业务学习
  2. 熟悉相关竞品软件，如西门子，罗克韦尔等
3.3.2 能力提升
  1. 阶段测试技能培训
  2. 提升研发自测意识
  3. 提升产品需求编写能力、增加产品原型、文档等输出
3.3.3 版本控制
  1. 以自然周为目标发布小版本（内部），发版前一天从 dev 合到 test 分支
  2. 以自然月为目标发布大版本（对外），发版前一周从 test 合到 master 分支
3.3.4 质量标准

  1. 测试准入标准：以接口自动化冒烟部分用例通过为标准，具体待定

  2. 测试覆盖率：以 Jacoco报告为标准，具体待定

  3. 软件发布标准：http://wiki.tcm.com:8082/x/2gVi

  4. 流程规范：缺少开发评审会议

3.4 探索
3.4.1 代码扫描

目前的代码扫描只是基于默认的规则，后期有时间可以探索自定义规则扫描

3.4.2 安全测试

结合项目实际需要研究安全扫描相关工具 kali linux

3.4.3 流量录制与回放

纯内网环境可能很难通过流量录制再回放，如果不行再研究下其他替代方案

3.4.4 混沌测试

K8S 下的微服务稳定性测试，暂定用阿里开源的 Chaosblade

3.4.5 契约测试

需要先实现契约，暂时没很好的想法，先提出，有时间研究

3.4.6 测试平台建设
